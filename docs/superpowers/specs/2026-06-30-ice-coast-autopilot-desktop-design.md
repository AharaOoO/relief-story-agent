# Relief Story Agent 冰海玻璃自动制片桌面端设计规格

> 状态：Approved
>
> 日期：2026-06-30
>
> 实施基线：`gemini/wp-101-contract-baseline@5699607`

## 1. 产品目标

Relief Story Agent 要成为一个普通创作者可以双击打开、快速发起单条或批量任务、持续观察十道工序并取得成片的 Windows 桌面客户端。用户不需要理解 Python、Vite、端口、JSON 或 ComfyUI 节点；这些内容只在高级设置和诊断中出现。

核心体验是一页式“海滩自动制片工作台”：首页保留清晰海滩视频；进入工作状态后仍能看到海岸氛围；创建任务后不跳回旧 UI；十道工序的状态全部来自真实后端。

## 2. 不可妥协的验收原则

1. 所有按钮点击后立即显示执行中、成功或失败反馈。
2. 预检不得清空故事输入、任务参数或提示词草稿。
3. 配置保存后必须被后端实际使用；仅写入前端状态不算完成。
4. 密钥不得回传明文到渲染进程，不得进入日志、Git、普通 JSON 或 localStorage。
5. RunningHub 国内站和国际站的模型列表、base URL、密钥和请求必须隔离。
6. 任务创建后停留在新工作台，以真实 run/batch timeline 驱动十步状态。
7. 旧 `AppShell`、旧管理台视觉和运行时 fixture 不得进入新产品路径。
8. 16:9 与 9:16 均可选，图像默认 2K，请求、校验和产物元数据保持一致。

## 3. 用户信息架构

顶层导航只保留：

- `控制台`：海滩首页、快速创建、最近任务和系统摘要。
- `自动执行`：十道工序工作台，是产品主界面。
- `任务队列`：批量任务、暂停、继续、取消和重试。
- `资产库`：四宫格、提示词包、视频和导出物。
- `高级设置`：以右侧抽屉打开，不离开当前任务。

端口、环境变量名、模型探针、工作流 JSON、输出目录、重试次数、日志和原始诊断只进入高级设置。普通用户只看到“已连接”“需要配置”“正在验证”等产品语言。

## 4. 首页与工作台布局

### 4.1 首页

首屏使用本地海滩视频，顶部是悬浮玻璃导航。视频底部通过 CSS mask 和同色渐变自然融入 `#F7F9FC` 内容背景，不出现硬切边。主标题使用现有衬线展示字体，正文使用稳定中文无衬线字体。

首页核心操作区包含：

- 一个合并输入框，接受灵感、创作要求、现有剧本或空白自动起题。
- 输入类型自动识别，并允许用户手动切换。
- 时长、16:9/9:16、任务数量、创作预设四个常用参数。
- `导入剧本`、`开始生成`、`高级设置`三个明确动作。
- 最近任务和系统连接状态的精简摘要。

### 4.2 自动执行工作台

采用三栏结构：

- 顶部：十道工序进度轨，显示序号、烹饪名称、技术名称和真实状态。
- 中部左侧：当前工序详情；第 1 至 6 步展开时显示模型选择和提示词模板。
- 中部右侧：当前输出、错误原因、建议动作和产物预览。
- 底部：任务输入摘要、暂停/继续/取消/重试和返回任务列表。

十道工序映射固定为：

| 序号 | 用户名称 | 后端 stage |
|---:|---|---|
| 1 | 备料 | `chief_screenwriter` |
| 2 | 慢炖 | `deepseek_polish` |
| 3 | 试味 | `quality_gate` |
| 4 | 配菜 | `gpt_prompt_writer` |
| 5 | 调味 | `gpt_prompt_audit` |
| 6 | 回锅 | `gpt_prompt_reviser` |
| 7 | 锁菜谱 | `final_prompts` |
| 8 | 出盘 | `four_grid_asset` |
| 9 | 打包 | `artifacts` |
| 10 | 出餐中 | `comfyui` |

## 5. 视觉系统

### 5.1 配色

- 页面背景：`#F7F9FC`
- 玻璃面：`rgba(255,255,255,0.64)`
- 主要文字：`#252832`
- 次要文字：`#7B8290`
- 冰蓝强调：`#6E9FFF`
- 成功：`#2FAE82`
- 警告：`#C98A26`
- 失败：`#D65C68`
- 边框：`rgba(123,145,184,0.20)`

页面不能变成单一蓝色主题。海滩沙白、冷灰、海蓝与少量绿色状态色共同形成层次。

### 5.2 字体

- 展示标题：沿用现有 `Instrument Serif` 风格，并提供宋体/思源宋体中文回退。
- 正文与控制：`Inter`, `Microsoft YaHei UI`, sans-serif。
- 不使用 viewport width 缩放字号，不使用负字距。
- 英文只用于模型名、技术标识和必要状态，不再作为无意义装饰。

### 5.3 间距与组件

- 内容最大宽度：1280px。
- 页面区块间距：48-80px。
- 卡片内边距：24-32px。
- 控件间距：12-20px。
- 顶部玻璃导航高 72px，圆角 30px。
- 功能卡圆角 20-24px，使用细冰蓝边框和双层轻阴影。
- 普通按钮圆角 14-18px；主 CTA 可使用胶囊形。
- 十步行使用固定高度和稳定网格，状态变化不能引起布局跳动。
- 卡片不嵌套卡片；页面区块保持通透，不堆叠装饰容器。

## 6. 背景资产

生成并本地打包三张 2K 位图：

1. 海浪高光与冰玻璃折射，用于首页视频过渡后的第一段。
2. 透明几何与水面焦散，用于工作台主体。
3. 柔和海岸线与蓝灰薄雾，用于资产库和队列底部。

三张图必须留出干净的上下边缘，通过 CSS mask、同色底和轻微视差拼接成连续背景。所有资产离线可用，禁止依赖远程 URL。

## 7. 普通创建与批量创建

输入模式：

- `auto`：输入为空，模型自动起题并确定内核、冲突和系列方向。
- `idea`：一句灵感。
- `requirements`：创作目标、受众、风格或限制。
- `script`：尽量保留现有剧情并转为导演分镜。
- `mixed`：已有内容和补充要求混合输入。

单任务和批量任务使用同一份默认配置。批量模式允许 1-20 条，支持逐条编辑和禁用。预检仅返回摘要和阻塞项，不重置表单。

## 8. 模型与提示词配置

第 1 至 6 步均有：启用状态、供应商模式、站点、模型、提示词模板、恢复默认值。第 6 步仅在审查失败时调用；通过时 UI 明确显示“无需回锅”。

### 8.1 RunningHub 国内站精选模型

- 总编剧：`qwen/qwen3.7-plus`, `qwen/qwen3.7-max`
- 改稿：`deepseek/deepseek-v4-flash`, `deepseek/deepseek-v4-pro`
- 质量门禁：`deepseek/deepseek-v4-pro`, `deepseek/deepseek-v4-flash`
- 分镜提示词：`qwen/qwen3.7-max`, `qwen/qwen3.7-plus`
- 审查：`deepseek/deepseek-v4-pro`, `deepseek/deepseek-v4-flash`
- 修订：`qwen/qwen3.7-plus`, `qwen/qwen3.7-max`

base URL 固定为 `https://llm.runninghub.cn/v1`。该标准模型端点只适用于 RunningHub 企业共享（SHARED / enterprise）API Key；普通消费级 key 应继续使用“普通 OpenAI-compatible 模式”或仅用于 G2/任务类 API。

### 8.2 RunningHub 国际站精选模型

- 总编剧：`google/gemini-3.5-flash`
- 改稿：`deepseek/deepseek-v4-pro`
- 质量门禁：`deepseek/deepseek-v4-pro`, `openai/gpt-5.5`
- 分镜提示词：`openai/gpt-5.5`
- 审查：`openai/gpt-5.4-mini`, `openai/gpt-5.5`
- 修订：`openai/gpt-5.4-mini`, `openai/gpt-5.5`

base URL 固定为 `https://llm.runninghub.ai/v1`。该标准模型端点只适用于 RunningHub 企业共享（SHARED / enterprise）API Key；普通消费级 key 应继续使用“普通 OpenAI-compatible 模式”或仅用于 G2/任务类 API。

### 8.3 普通 OpenAI-compatible 模式

每个阶段可单独设置 base URL、model 和对应密钥引用。该模式不复用 RunningHub 站点切换，也不展示 RunningHub 精选模型列表。

## 9. 高级设置抽屉

宽度约 600px，包含六个标签页：

1. `模型与密钥`：模式、站点、密钥状态、测试连接、六阶段模型。
2. `提示词模板`：六阶段编辑、校验、保存、恢复默认。
3. `ComfyUI`：endpoint、工作流拖拽/选择、分析结果、测试连接。
4. `图像`：G2/兼容生图 provider、16:9/9:16、默认 2K。
5. `执行与存储`：输出目录、队列、重试和并发。
6. `诊断`：端口、版本、日志、原始 JSON 和导出诊断包。

抽屉关闭后不丢失未提交编辑；保存按钮显示阶段化反馈：正在校验、正在保存、正在重启后端、已生效或失败。

## 10. 后端增量设计

保留当前调度器、持久化、ComfyUI、恢复和产物能力，补齐以下契约：

- Prompt Profile CRUD、克隆、恢复默认和版本快照 API。
- 第 3 步 LLM 质量报告与本地硬规则合并；硬规则始终拥有否决权。
- RunningHub enterprise LLM provider，使用 OpenAI-compatible chat completion，按站点隔离，并在 UI/错误信息中明确企业共享 key 要求。
- RunningHub G2 task provider，创建任务、轮询、下载并记录 2K/比例元数据。
- 运行请求显式携带六阶段模型绑定、Prompt Profile、ComfyUI 和图像配置。
- 增量事件游标，供 UI 无重复轮询当前 run/batch。

## 11. Electron 桌面运行时

Electron 负责：

- 启动、监控和关闭 Python sidecar。
- 选择剧本、工作流、输出目录和打开日志/产物目录。
- 用 `safeStorage` 保存密钥，仅向渲染进程返回 `configured` 与掩码摘要。
- 保存非敏感配置到 AppData，并以原子写入保证重启后可恢复。
- 配置变更后受控重启后端并重新握手。
- 收集 sidecar 日志和启动错误，不再丢弃 stdio。
- 在端口冲突时选择可用端口并通过 handshake 注入前端。

## 12. 错误、恢复与可访问性

- 每个异步动作具有 pending/success/error 状态和 `aria-live` 文案。
- 错误卡显示用户可执行的下一步，不默认暴露堆栈。
- 高级设置抽屉支持 Esc、焦点圈定和键盘导航。
- `prefers-reduced-motion` 下禁用视差和大幅动画。
- 后端中断后 UI 保留草稿和 run id，重连后恢复轮询。

## 13. 完成定义

只有以下证据全部成立才算完成：

- 后端测试、前端测试、类型检查、lint 和 production build 全部通过。
- Electron 主进程/preload 测试与开发启动通过，且无残留进程。
- 1280x800、1440x900、1920x1080 与 440x900 截图无重叠、截断和空白画布。
- 单任务和三任务批次均能从新 UI 创建并观察真实十步状态。
- 工作流拖拽/选择、密钥保存、重启后恢复和测试连接均有真实证据。
- 国内/国际 RunningHub 下拉列表不会串台。
- 未提交密钥、`.env`、用户剧本、机器私有路径或诊断数据。
